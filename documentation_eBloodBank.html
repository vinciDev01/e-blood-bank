Documentation eBloodBank

Systeme de gestion de banque de sang developpe avec Django 4.2. Ce document explique l'architecture du projet, ses fonctionnalites et les procedures a suivre pour l'utiliser et le maintenir.

1. Presentation du projet

eBloodBank est une application web qui permet de gerer les dons de sang, le stock de poches de sang, et les demandes provenant des services medicaux (hopitaux, cliniques, etc.).

Que fait cette application ?

  • Gerer les donneurs : enregistrer les donneurs, suivre leurs dons et leur groupe sanguin.
  • Gerer le stock de sang : ajouter des poches de sang, suivre la disponibilite par groupe sanguin, generer des codes QR.
  • Traiter les demandes : les services medicaux font des demandes de sang, la banque de sang les accepte ou les refuse.
  • Gerer les utilisateurs : inscription, connexion, verification par email, acces selon le role.

Les 4 types d'utilisateurs

RoleCodeDescription
Utilisateur generique generic Visiteur inscrit, peut voir les hopitaux et faire des demandes basiques.
Service medical medical Hopital ou clinique, peut faire des demandes de sang et gerer ses patients.
Donneur donor Donneur de sang enregistre, a acces a son espace personnel.
Banque de sang blood_bank Gere le stock, accepte/refuse les demandes, enregistre les poches.

2. Architecture generale

Le projet suit l'architecture classique de Django : il est divise en applications (apps), chacune ayant une responsabilite precise.

eBloodBank/

Configuration du projet (settings, URLs racine, email)

_auth/

Authentification, modeles utilisateurs, inscription, connexion

bankDeSang/

Gestion des poches de sang, stock, dons

serviceMedicaux/

Demandes de sang, patients, stock par service

donneur/

Espace donneur (vues uniquement, modeles dans _auth)

frontend/

Pages publiques, templates, fichiers statiques

Point important : Tous les modeles lies aux utilisateurs (Donneur, BanqueDeSang, ServiceMedicaux, Utilisateur) sont definis dans _auth/models.py, pas dans leurs applications respectives. C'est un choix de conception pour centraliser la gestion des profils.

3. Detail des applications

3.1 Application _auth (Authentification)

C'est l'application la plus importante. Elle contient :

  • Le modele CustomUser qui etend le systeme d'authentification de Django avec un champ role.
  • Les modeles de profil : Utilisateur, Donneur, ServiceMedicaux, BanqueDeSang.
  • Les vues d'inscription, connexion, deconnexion et activation par email.
  • Un backend d'authentification personnalise (CustomUserBackend).
  • Un generateur de token pour la verification par email (token.py).

3.2 Application bankDeSang (Banque de sang)

Gere tout ce qui concerne le stock de sang :

  • PocheDeSang : une poche de sang avec son matricule, groupe sanguin, date d'expiration et code QR.
  • DonDeSang : enregistrement d'un don associe a un donneur et une poche.
  • StockDeSang : suivi du nombre de poches par groupe sanguin.
  • Vues pour ajouter des poches, voir le stock, accepter/refuser les demandes.

3.3 Application serviceMedicaux (Services medicaux)

Gere les demandes de sang des hopitaux :

  • Patient : informations du patient qui a besoin de sang.
  • DemandeDeSang : la demande elle-meme, avec les groupes sanguins demandes, les quantites, l'urgence et l'etat.
  • Stock_de_sang : stock de sang recu par un service medical.

3.4 Application donneur (Donneur)

Contient uniquement les vues pour l'espace donneur. Les modeles sont dans _auth/models.py.

3.5 Application frontend (Pages publiques)

Les pages visibles par tous : accueil, liste des hopitaux, detail d'un hopital. Contient aussi tous les templates HTML du projet.

6. Systeme d'authentification

6.1 Modele utilisateur personnalise

Django utilise normalement le modele User par defaut. Ce projet le remplace par CustomUser (defini dans _auth/models.py) qui ajoute un champ role. Ce role determine ce que l'utilisateur peut faire et ou il est redirige apres connexion.

6.2 Backend d'authentification

Le fichier _auth/backends.py contient CustomUserBackend, un backend qui authentifie par nom d'utilisateur. Il est configure dans settings.py aux cotes du backend par defaut de Django.

6.3 Controle d'acces par role

Le fichier decorateurs.py (a la racine du projet) contient le decorateur @check_role(role). Ce decorateur est place sur les vues pour restreindre l'acces :

@login_required
@check_role('blood_bank')
def accueilBankDeSang(request):
    # Seul un utilisateur avec le role 'blood_bank' peut acceder ici
    ...

Si le role ne correspond pas, l'utilisateur est redirige vers la page d'accueil avec un message d'erreur.

6.4 Verification par email

A l'inscription, le compte est cree comme inactif. Un email contenant un lien d'activation est envoye. Ce lien contient un identifiant encode (uidb64) et un token genere par token.py. Quand l'utilisateur clique sur le lien, son compte est active.

7. Fonctionnalites : Banque de sang

L'utilisateur avec le role blood_bank a acces a ces fonctionnalites :

7.1 Tableau de bord

Affiche le nombre total de poches disponibles et le nombre de demandes en attente.

7.2 Gestion du stock

Permet d'ajouter de nouvelles poches de sang. Pour chaque poche, on renseigne :

  • Le donneur (selectionne dans la liste)
  • Le matricule (identifiant unique)
  • Le type de produit (sang total, globules rouges, plasma, plaquettes)
  • Le groupe sanguin
  • La date de prelevement

A la creation, le systeme calcule automatiquement la date d'expiration (42 jours) et genere un code QR contenant les informations de la poche.

7.3 Traitement des demandes

La banque de sang voit toutes les demandes en attente et peut :

  • Accepter : selectionner les poches disponibles a attribuer, elles passent en etat en_transition.
  • Refuser : rejeter une demande pour un groupe sanguin specifique.

7.4 Statistiques

Un tableau de bord avec des statistiques sur le stock et les demandes traitees.

8. Fonctionnalites : Service medical

8.1 Faire une demande de sang

Un service medical peut creer une demande en specifiant :

  • Un ou plusieurs groupes sanguins necessaires
  • Le nombre de poches pour chaque groupe
  • Le type de produit souhaite
  • Le niveau d'urgence : Immediate, 24 heures, ou Non urgent
  • Le motif : Chirurgie, Accident, Maladie, Autre
  • Les informations du patient
Champs JSON : La demande utilise des champs JSON pour permettre de demander plusieurs groupes sanguins dans une seule demande. Par exemple, une demande peut concerner 3 poches de A+ et 2 poches de O- simultanement.

8.2 Suivi des demandes

Le service medical peut voir toutes ses demandes et leur etat :

  • En attente : la demande a ete envoyee
  • Approuvee : la banque a accepte
  • Rejetee : la banque a refuse
  • Completee : les poches ont ete recues

8.3 Reception des poches

Quand les poches arrivent, le service medical confirme la reception. Les poches sont alors ajoutees a son propre stock (Stock_de_sang) et retirees du stock de la banque de sang.

9. Fonctionnalites : Espace donneur

L'espace donneur est actuellement minimal. Il contient :

  • Un tableau de bord d'accueil (accueilDonneur)
  • Une vue pour lister les donneurs (listeDonneur)

Le profil du donneur (nom, groupe sanguin, numero de donneur, etc.) est gere via le modele Donneur dans _auth/models.py.

10. Pages publiques (Frontend)

Ces pages sont accessibles a tous les visiteurs :

  • Accueil (/) : page d'accueil du site.
  • Liste des hopitaux (/listeHopitaux) : liste de tous les services medicaux inscrits.
  • Detail hopital (/hopital/<id>) : informations detaillees sur un hopital.
  • Mes demandes (/listeDemandeDeSang) : les demandes de sang liees a l'utilisateur connecte.

11. Procedure d'installation

Prerequis

  • Python 3.8 ou superieur
  • MySQL installe et en cours d'execution
  • pip (gestionnaire de paquets Python)

Etapes

Etape 1 : Creer la base de donnees MySQL

mysql -u root -p
CREATE DATABASE ebloodbank CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
EXIT;
Identifiants par defaut : La configuration actuelle utilise root:root sur localhost:3306. Modifiez eBloodBank/settings.py si vos identifiants sont differents.

Etape 2 : Creer un environnement virtuel

python -m venv .venv
source .venv/bin/activate    # Linux / macOS
# .venv\Scripts\activate     # Windows

Etape 3 : Installer les dependances

pip install -r requirements.txt

Etape 4 : Appliquer les migrations

python manage.py makemigrations
python manage.py migrate

Etape 5 : Creer un super utilisateur (optionnel)

python manage.py createsuperuser

Etape 6 : Lancer le serveur

python manage.py runserver

Le site est accessible sur http://127.0.0.1:8000/

Etape 7 : Configurer l'envoi d'emails (optionnel)

Modifiez le fichier eBloodBank/info.py avec vos identifiants Gmail pour activer l'envoi d'emails de verification.

Securite : Le fichier eBloodBank/info.py contient des identifiants email. Ne le partagez jamais publiquement. En production, utilisez des variables d'environnement.

12. Procedure d'inscription

Inscription d'un utilisateur generique

1. Aller sur /_auth/register/ → 2. Remplir le formulaire → 3. Email d'activation envoye → 4. Cliquer sur le lien → 5. Compte actif

Champs requis : prenom, nom, email, mot de passe, confirmation du mot de passe.

Le compte est cree avec le role generic et reste inactif jusqu'a la verification par email.

Inscription d'un service medical

1. Aller sur /_auth/inscriptionServiceMedicaux/ → 2. Remplir les informations → 3. Fournir le certificat → 4. Compte cree (role medical)

Champs requis : nom de l'etablissement, type, responsable, adresse complete, email, telephone, numero de licence, numero d'enregistrement, certificat d'enregistrement.

13. Procedure de connexion

1. Aller sur /_auth/login/ → 2. Saisir email + mot de passe → 3. Redirection selon le role

14. Procedure de demande de sang

C'est le flux principal de l'application. Voici comment il fonctionne du debut a la fin :

Etape 1 : Creation de la demande

Le service medical remplit le formulaire a /serviceMedicaux/faireDemandeDeSang/ :

  • Selectionner un ou plusieurs groupes sanguins
  • Indiquer le nombre de poches pour chaque groupe
  • Choisir le type de produit
  • Indiquer l'urgence et le motif
  • Renseigner les informations du patient

La demande est creee avec l'etat En attente.

Etape 2 : Traitement par la banque de sang

La banque de sang voit la demande dans sa liste (/bankDeSang/listeDemandesDeSang/). Elle peut :

Accepter la demande :

  1. Voir les poches disponibles pour le groupe sanguin demande
  2. Selectionner les poches a attribuer
  3. Les poches selectionnees passent en etat en_transition = True et est_disponible = False
  4. L'etat de la demande (pour ce groupe) passe a Approuvee
  5. Le stock de la banque est mis a jour

Refuser la demande :

  1. L'etat du groupe sanguin concerne passe a Rejetee

Etape 3 : Reception par le service medical

  1. Le service medical confirme la reception a /serviceMedicaux/recevoir_poches/
  2. Les poches sont ajoutees au stock du service medical (Stock_de_sang)
  3. Les poches ne sont plus en_transition
  4. L'etat passe a Completee

Schema du flux complet

Service medical cree la demande → Banque recoit la demande → Banque accepte / refuse → Service confirme reception → Stock mis a jour

15. Procedure de gestion du stock

Ajouter une poche de sang

  1. Se connecter en tant que blood_bank
  2. Aller sur /bankDeSang/gestionStock/
  3. Remplir le formulaire : donneur, matricule, type de produit, groupe sanguin, date de prelevement
  4. Le systeme cree la poche avec :
    • Date d'expiration = date de prelevement + 42 jours
    • Code QR genere automatiquement (sauvegarde dans media/QR code/)
    • Etat : disponible = True, en transition = False
  5. Le stock global (StockDeSang) est incremente pour le groupe sanguin correspondant

Consulter le stock

  • Vue globale : le tableau de bord affiche le total par groupe sanguin
  • Vue detaillee : /bankDeSang/detailStock/<id>/ montre toutes les poches d'un groupe sanguin

Cycle de vie d'une poche

Creee (disponible) → Attribuee (en transition) → Recue par le service → Dans le stock du service

17. Securite

Mesures en place

MesureImplementation

Controle d'accesDecorateur @check_role() sur les vues
Verification emailToken d'activation a usage unique
Protection CSRFMiddleware CsrfViewMiddleware actif

Points d'attention :
  • Le fichier info.py contient des identifiants en clair — a remplacer par des variables d'environnement en production.
  • DEBUG = True dans settings.py — a desactiver en production.
  • ALLOWED_HOSTS = [] — a configurer en production.
  • Les identifiants MySQL (root:root) sont codes en dur — a externaliser.

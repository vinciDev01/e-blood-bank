1. Securite et Credentials exposes

Critique SECRET_KEY Django expose en dur
eBloodBank/settings.py:23
La cle secrete Django est codee en dur dans le code source : 'django-insecure-ya@s2s6hto^...'. Cette cle signe les sessions, les tokens CSRF et les cookies. Si elle est compromise, toute la securite de l'application est en danger.
Correction : Utiliser une variable d'environnement : SECRET_KEY = os.environ.get('DJANGO_SECRET_KEY') et generer une nouvelle cle avec django.core.management.utils.get_random_secret_key().
Critique Credentials email Gmail exposes
eBloodBank/info.py:1-8
Le fichier info.py contient en clair l'adresse email (juniorbgj20@gmail.com) et le mot de passe d'application (bpkh mbsr syzh sygb). Ce fichier est versionne dans le code source.
Correction : Deplacer dans un fichier .env non versionne, utiliser python-dotenv ou django-environ pour charger les variables. Revoquer immediatement le mot de passe d'application Gmail actuel et en generer un nouveau.
Critique Mot de passe base de donnees faible et expose
eBloodBank/settings.py:109-110
La base de donnees MySQL utilise l'utilisateur root avec le mot de passe 12345678, code en dur dans settings.py.
Correction : Creer un utilisateur MySQL dedie avec des privileges limites. Stocker les credentials dans des variables d'environnement. Ne jamais utiliser root en production.
Critique DEBUG = True en mode production
eBloodBank/settings.py:42
DEBUG = True expose les traces d'erreur completes, les requetes SQL et la configuration de l'application aux utilisateurs.
Correction : DEBUG = os.environ.get('DJANGO_DEBUG', 'False') == 'True'
Critique ALLOWED_HOSTS vide
eBloodBank/settings.py:44
ALLOWED_HOSTS = [] bloquera toute requete en production (sauf localhost avec DEBUG=True).
Correction : ALLOWED_HOSTS = os.environ.get('ALLOWED_HOSTS', 'localhost,127.0.0.1').split(',')
Haute Pas de fichier .gitignore
Racine du projet
Aucun fichier .gitignore n'a ete detecte. Les fichiers sensibles (info.py, db.sqlite3, .env, __pycache__, media/) risquent d'etre commites.
Correction : Creer un .gitignore avec les exclusions standard Django : *.pyc, __pycache__/, .env, db.sqlite3, media/, *.save.

2. Application _auth - Authentification et Modeles

Bugs critiques dans les modeles

Critique Fonction numero_de_donneur() utilise Donneur.nom au lieu de instance.nom
_auth/models.py:17
La fonction accede a Donneur.nom (attribut de classe, inexistant) au lieu de instance.nom. Cela provoque un AttributeError a chaque creation d'un donneur.
# ACTUEL (bugge)
def numero_de_donneur():
    return f"{date.today().year}{'ebb'}{Donneur.nom}{Donneur.objects.count() + 1:04d}"

# CORRECTION
def numero_de_donneur(instance=None):
    nom = instance.nom if instance else ''
    return f"{date.today().year}ebb{nom}{Donneur.objects.count() + 1:04d}"
Critique renomer_image() incompatible avec tous les modeles
_auth/models.py:9-14
La fonction accede a instance.nom et instance.prenom, mais les modeles BanqueDeSang et ServiceMedicaux n'ont pas ces champs. L'upload de photo de profil plantera pour ces modeles.
Correction : Utiliser hasattr() pour verifier les champs ou utiliser instance.user.username comme identifiant universel pour le nommage des images.
Haute ServiceMedicaux definit USERNAME_FIELD et REQUIRED_FIELDS
_auth/models.py:83-97
Ces attributs sont propres aux modeles d'authentification (AbstractUser). ServiceMedicaux est un modele de profil, pas un modele d'utilisateur. Cela peut creer des conflits inattendus.
Correction : Supprimer USERNAME_FIELD et REQUIRED_FIELDS du modele ServiceMedicaux.
Moyenne Redondance du champ email entre CustomUser et Utilisateur
_auth/models.py
CustomUser.email et Utilisateur.email existent en parallele, creant une duplication de donnees et des risques de desynchronisation.
Correction : Supprimer le champ email de Utilisateur et utiliser utilisateur.user.email partout.

Bugs critiques dans les vues

Critique resetPasswordEmail/Confirm utilise User au lieu de CustomUser
_auth/views.py:158, 191
Les vues de reinitialisation de mot de passe utilisent User.objects.filter() et User.objects.get() au lieu de CustomUser. Le modele utilisateur est _auth.CustomUser, donc ces requetes ne trouveront jamais aucun utilisateur.
# ACTUEL (bugge)
User.objects.filter(email=email)
User.objects.get(pk=uid)

# CORRECTION
CustomUser.objects.filter(email=email)
CustomUser.objects.get(pk=uid)
Critique Redirect casse dans resetPasswordConfirm
_auth/views.py:206
return redirect('resetPasswordConfirm', uidb64, token) ne contient pas le namespace _auth:. Provoque une erreur NoReverseMatch.
Correction : return redirect('_auth:resetPasswordConfirm', uidb64, token)
Critique 4 vues password reset sans URL pattern
_auth/urls.py
Les vues resetPassword, resetPasswordEmail, resetPasswordConfirm et forgotPassword sont definies dans views.py mais n'ont aucun chemin dans urls.py. La fonctionnalite de reinitialisation de mot de passe est totalement inaccessible.
# Ajouter dans _auth/urls.py :
path('resetPassword/', views.resetPassword, name='resetPassword'),
path('resetPasswordEmail/', views.resetPasswordEmail, name='resetPasswordEmail'),
path('resetPasswordConfirm/<uidb64>/<token>/', views.resetPasswordConfirm, name='resetPasswordConfirm'),
path('forgotPassword/', views.forgotPassword, name='forgotPassword'),
Haute logoutServiceMedicaux() n'effectue pas la deconnexion
_auth/views.py:292-294
La vue affiche un message de succes et rend un template, mais n'appelle jamais logout(request). La session utilisateur reste active.
Correction : Ajouter logout(request) avant le rendu du template.
Haute 4 templates email manquants
_auth/views.py:153, 162, 207, 213
Les templates suivants sont references mais n'existent pas : auth/users/resetPasswordMail.html, auth/users/resetPasswordConfirm.html, auth/users/forgotPassword.html, auth/users/resetPassword.html.
Correction : Creer les 4 templates dans _auth/templates/auth/users/.
Moyenne try-except masque les erreurs dans logIn
_auth/views.py:88
Un try-except AttributeError masque les vraies erreurs. Si user est None, user.role leve AttributeError mais d'autres erreurs du meme type seront egalement englouties.
Correction : Remplacer par if user is not None: puis acceder a user.role.

Incoherences des roles

Moyenne roles.py incoherent avec CustomUser.ROLE_CHOICES
_auth/roles.py vs _auth/models.py
roles.py definit 'banque_de_sang' mais CustomUser.ROLE_CHOICES utilise 'blood_bank'. Les constantes de roles.py ne sont jamais importees (import commente). Deux systemes de roles coexistent sans aucun lien.
Correction : Supprimer roles.py ou l'aligner avec ROLE_CHOICES et l'utiliser comme source unique de verite.
Basse admin.py vide - aucun modele enregistre
_auth/admin.py
Aucun des 5 modeles (CustomUser, Utilisateur, Donneur, BanqueDeSang, ServiceMedicaux) n'est enregistre dans l'interface d'administration Django.
Correction : Enregistrer les modeles avec admin.site.register() et creer des classes ModelAdmin appropriees.
Basse Import duplique de models
_auth/models.py:1-2
from django.db import models apparait deux fois.
Correction : Supprimer la ligne dupliquee.

3. Application bankDeSang - Gestion des poches de sang

Bugs critiques

Critique notification() utilise le champ inexistant 'statut'
bankDeSang/views.py:34
Le filtre DemandeDeSang.objects.filter(statut='En attente') utilise statut mais le champ s'appelle etat. Provoque un FieldError a chaque appel.
Correction : Remplacer statut par etat.
Critique Filtrage JSONField incorrect dans listeDemandesDeSang
bankDeSang/views.py:128
PocheDeSang.objects.filter(groupe_sanguin=demande.groupe_sanguin) compare un CharField avec un JSONField (dictionnaire). Le filtre ne retournera jamais de resultats.
Correction : Iterer sur les cles du JSONField et filtrer par chaque groupe sanguin individuellement.
Critique Etat '1/2 Approuvee' absent des choix du modele
bankDeSang/views.py:116
Le filtre utilise '1/2 Approuvee' comme valeur d'etat, mais DemandeDeSang.etat_choices ne contient que : En attente, Approuvee, Rejetee, Completee. Les demandes partiellement approuvees ne seront jamais trouvees.
Correction : Ajouter ('1/2 Approuvee', '1/2 Approuvee') dans les choix du modele ou revoir la logique de filtrage.

Securite et logique

Haute poches_disponibles() retourne les poches de TOUTES les banques
bankDeSang/views.py:217
Le filtre PocheDeSang.objects.filter(groupe_sanguin=..., est_disponible=True) ne filtre pas par banque de sang. Une banque peut voir et allouer les poches des autres banques.
Correction : Ajouter bank_de_sang=request.user.banque_de_sang au filtre.
Haute notification() sans decorateur d'authentification
bankDeSang/views.py:33
La vue n'a ni @login_required ni @check_role. N'importe qui peut acceder aux notifications des demandes de sang.
Correction : Ajouter @login_required et @check_role('blood_bank').
Haute Pas de try-except pour Donneur.objects.get() dans gestionStock
bankDeSang/views.py:74
Si l'ID du donneur est invalide, Donneur.objects.get(id=donneur_id) leve DoesNotExist et provoque une erreur 500.
Correction : Utiliser get_object_or_404(Donneur, id=donneur_id).
Moyenne Import non standard de settings
bankDeSang/models.py:10
from eBloodBank import settings au lieu de from django.conf import settings. Peut echouer dans certaines configurations.
Correction : Utiliser from django.conf import settings.
Moyenne Champ en_transition jamais remis a False
bankDeSang/views.py (accepter_demande)
Le champ en_transition est mis a True lors de l'acceptation mais n'est jamais remis a False apres reception. Les poches restent en etat de transition indefiniment.
Correction : Mettre en_transition=False dans la vue recevoir_poches apres reception.
Basse Instructions print() de debug laissees dans le code
bankDeSang/views.py:35, 43, 44, 140-145
Plusieurs print() de debug sont presents dans le code de production.
Correction : Supprimer tous les print() ou les remplacer par logging.debug().
Basse login_url code en dur dans tous les decorateurs
bankDeSang/views.py (multiple)
Chaque @login_required utilise login_url='/_auth/authentification/' en dur au lieu de settings.LOGIN_URL.
Correction : Definir LOGIN_URL = '/_auth/login/' dans settings.py et omettre le parametre login_url.

4. Application serviceMedicaux

Bugs critiques

Critique Reference a StockDeSang supprime dans recevoir_poches
serviceMedicaux/views.py:232
StockDeSang.enregistrer_stock(poche, -1) reference une classe supprimee dans la migration 0006. Provoque un NameError a l'execution. Seul Stock_de_sang existe encore.
# ACTUEL (bugge) - StockDeSang n'existe plus
StockDeSang.enregistrer_stock(poche, -1)  # NameError!
Stock_de_sang.enregistrer_stock(poche, 1)

# CORRECTION - utiliser uniquement Stock_de_sang
# Retirer du stock de la banque (via bankDeSang.models.StockDeSang si necessaire)
Stock_de_sang.enregistrer_stock(poche, 1)
Critique getToutesDemande() utilise .get() au lieu de .filter()
serviceMedicaux/views.py:206
DemandeDeSang.objects.get(etat='En attente') leve MultipleObjectsReturned des qu'il y a plus d'une demande en attente.
Correction : Remplacer .get() par .filter().
Critique Comparaison semantiquement incorrecte dans recevoir_poches
serviceMedicaux/views.py:218-222
Le code compare des groupes sanguins (ex: "A+") avec des matricules de poches (ex: "2024ebb0001"). La condition if groupe in poches ne sera jamais vraie, car ce sont des types de donnees differents.
Correction : Revoir la logique pour verifier si le groupe sanguin a des poches associees dans nombre_poches_allouees.

Securite et authentification

Haute 4 vues sans @login_required
serviceMedicaux/views.py:26, 54, 205, 211
Les vues mesDemandesDeSang, listeDemandeDeSang, getToutesDemande et recevoir_poches n'ont pas de decorateur d'authentification. N'importe quel utilisateur anonyme peut acceder aux demandes de sang et recevoir des poches.
Correction : Ajouter @login_required et @check_role('medical') a chaque vue.
Haute Pas de gestion d'erreur dans recevoir_poches
serviceMedicaux/views.py:215, 227
Les appels .get() sans try-except provoqueront des erreurs 500 si l'ID de demande ou le matricule de poche est invalide.
Correction : Utiliser get_object_or_404().

Structure des donnees JSON

Moyenne Structure JSONField incoherente selon le role
serviceMedicaux/views.py:72-119
La vue faireDemandeDeSang cree des structures JSON differentes selon le role : {email: [liste]} pour 'medical', {"groupe_sanguin": valeur} pour 'donor', {user_email: valeur} pour 'generic'. Les vues de lecture supposent une structure unique.
Correction : Normaliser la structure JSON. Utiliser un format uniforme comme {"groupes": ["A+", "B-"], "quantites": [2, 1]}.
Moyenne Acces aux JSONField sans gestion de KeyError
serviceMedicaux/views.py:33-34
demande.groupe_sanguin[service_medical.email] leve KeyError si l'email n'est pas une cle du dictionnaire JSON.
Correction : Utiliser .get(service_medical.email, []).
Basse Import duplique dans models.py
serviceMedicaux/models.py:2, 4
from _auth.models import * est importe deux fois.
Correction : Supprimer la ligne dupliquee.
Basse Import inutilise de check_role
serviceMedicaux/views.py:11
from decorateurs import check_role est importe mais jamais utilise dans ce fichier.
Correction : Supprimer l'import ou l'utiliser sur les vues qui en ont besoin.

8. Incoherences architecturales

Moyenne Modeles repartis de maniere incoherente entre les apps

Tous les modeles lies aux utilisateurs (Donneur, BanqueDeSang, ServiceMedicaux, Utilisateur) sont dans _auth/models.py, pas dans leurs apps respectives. Cela cree des dependances circulaires et rend l'architecture confuse :

  • donneur/models.py est entierement commente (60 lignes de code mort)
  • frontend/models.py est vide avec un commentaire de modele planifie
  • bankDeSang et serviceMedicaux importent tout depuis _auth.models
Moyenne Aucune application n'utilise de Django Forms

Aucun fichier forms.py n'existe dans tout le projet. Toute la validation est manuelle via request.POST.get() :

  • Pas de validation de type ou de format
  • Pas de messages d'erreur structures
  • Pas de nettoyage des donnees (sanitization)
  • Duplication de la logique de validation entre les vues
Correction : Creer des ModelForm pour chaque modele necesitant un formulaire. Centraliser la validation.
Moyenne Aucun modele enregistre dans l'admin Django
Les 5 fichiers admin.py du projet sont tous vides. Aucun modele n'est administrable via l'interface d'administration Django, rendant la gestion des donnees impossible sans acces direct a la base de donnees.
Correction : Enregistrer au minimum CustomUser, Donneur, BanqueDeSang, ServiceMedicaux, PocheDeSang, DemandeDeSang dans l'admin.
Basse Convention de nommage mixte (camelCase vs snake_case)
TypecamelCasesnake_case
VuesaccueilBankDeSangaccepter_demande
URLslisteDemandesDeSangpoches_disponibles
ModelesStockDeSangStock_de_sang
ChampsserviceMedicauxservice_medical
Correction : Adopter la convention PEP 8 : snake_case pour les fonctions, variables et noms de fichiers ; PascalCase pour les classes.
Basse Aucun test ecrit
Tous les fichiers tests.py sont vides. Aucun test unitaire ou d'integration n'existe pour valider le fonctionnement des modeles, vues ou formulaires.
Correction : Ecrire des tests pour les fonctionnalites critiques : creation de donneur, demande de sang, gestion de stock, authentification.
